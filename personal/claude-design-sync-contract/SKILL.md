# Claude Design Sync Contract

## Core Principle

Claude Design should sync from the same committed public design-system surface that the application can maintain. The default job is to repair the `.design-sync` contract, not to redesign the app's component architecture.

Do not make `.design-sync` depend on generated, gitignored, prototype-only, or deep source entry files as the long-term contract. Treat current repo files as the source of truth; Claude Design and `/design-sync` behavior may change, so do not assume every project has the same generated layout.

Target chain:

```text
real React source
-> committed design-system barrel
-> app, feature, shared, and pattern code importing the same public surface
-> .design-sync/config.json entry
-> generated Claude Design package or preview surface
-> .design-sync/previews importing only that public surface
```

## When To Use

Use this skill when the user asks to:

- set up or repair Claude Code `/design-sync`;
- decide what `.design-sync/config.json` should point at;
- unify Claude Design previews with a real React component library;
- remove deep source imports from `.design-sync/previews`;
- decide whether app or feature code should import primitives through a barrel;
- decide where primitives and composed UI patterns should live;
- clean stale `.design-sync/NOTES.md`, `.design-sync/conventions.md`, or prototype references.

If the task is a one-off Claude Design artifact handoff, zip/HTML import, repo artifact archive, or Codex implementation, use `codex-claude-design-handoff` instead. Use this skill only for the design-system sync contract and repo conventions.

## Inspect First

Read existing files before changing anything. Do not create missing `.design-sync` files just because this list names them.

- `.design-sync/config.json`
- `.design-sync/NOTES.md`
- `.design-sync/conventions.md`
- `.design-sync/previews/**`
- `.ds-sync/**` when present, only to identify generated local output
- `package.json`
- `tsconfig.json` or Vite path alias config
- existing component directories such as `src/components/ui`, `src/components/shared`, `src/components/patterns`, `src/ui/primitives`, `src/ui/components`
- existing public barrels such as `src/components/ui/index.ts`, `src/design-system/index.ts`, `src/ui/index.ts`, or package `exports`
- app, feature, shared, and pattern imports that consume UI primitives

Use `rg` to confirm actual import paths and usage. Do not infer the source of truth from directory names alone.

## Contract Files

Treat these as maintainable source contracts when they exist:

- `.design-sync/config.json`
- `.design-sync/NOTES.md`
- `.design-sync/conventions.md`
- `.design-sync/previews/**`

Treat `.ds-sync/` as local generated output when it exists. It can usually remain untracked, be deleted, or be regenerated by the tool. Do not point durable config or preview imports at `.ds-sync/` files.

Some projects may also have generated or ignored files under `.design-sync`. Check `git status`, `.gitignore`, and file contents before deciding authority. A file is authoritative only when the repo treats it as a committed source contract.

## Public Surface Contract

Prefer one real, committed barrel as the public UI surface. The barrel should serve real app consumers and Claude Design previews; otherwise it is only a tool shim and should not be documented as a public API.

- Primitive-only systems can use `src/components/ui/index.ts` or the repo's existing primitive barrel.
- Mixed systems should expose a top-level barrel such as `src/design-system/index.ts` or `src/ui/index.ts`.
- Published package workspaces can use the package entrypoint or package export that app code already consumes.

The barrel should export both foundations and allowed higher-level UI:

```text
src/components/ui/index.ts
  Button, Badge, Input, Dialog, Tabs, Select

src/components/patterns/index.ts
  ProjectCard, WorkspaceHub, HeaderShell

src/design-system/index.ts
  re-export primitives and patterns intended for Claude Design
```

Do not expose feature-private screens, route components, data-fetching hooks, or app-only providers through the design-system barrel unless the repo already treats them as public UI contracts.

If app code has many deep imports and the repo wants a public UI API, recommend a scoped migration plan instead of rewriting everything incidentally. If the repo deliberately follows pure shadcn file-level imports and does not want a public UI API, remove or weaken unused barrels rather than leaving a false boundary.

## App Import Contract

When the goal is a public UI API or a Claude Design source-of-truth surface, app code should converge on the same public names as previews.

Use this boundary:

- app, route, feature, shared, and pattern code imports primitives from the public barrel, such as `@/ui`, `@/components/ui`, `@scope/ui`, or the package name;
- `src/components/ui/*`, `src/ui/primitives/*`, or equivalent primitive internals may keep file-level imports to avoid barrel self-imports and cycles;
- tightly scoped implementation files may deep import when the repo already treats them as internals, but document the exception;
- `.design-sync/previews/**` imports the package surface produced by `/design-sync`, not app-local deep paths.

Allowed app shape:

```tsx
import { Button, Dialog, Input } from "@/components/ui";
```

Avoid in app, feature, shared, and pattern layers when the repo has a public UI barrel:

```tsx
import { Button } from "@/components/ui/button";
import { Dialog } from "@/components/ui/dialog";
```

Do not rewrite primitive-directory internals to import from the barrel:

```tsx
// inside src/components/ui/column-filter-select.tsx
import { DropdownMenu } from "@/components/ui/dropdown-menu";
```

## Repo Shape Rules

Classify the repo before recommending structure.

### Monorepo Or App-Local Design System

Typical source-of-truth paths:

- `apps/web/src/ui/index.ts`
- `apps/<app>/src/components/ui/index.ts`
- `packages/ui/src/index.ts`
- package `exports` for an internal UI workspace

Rules:

- run `/design-sync` from the package or app root that owns the canonical UI source, or point config at that source explicitly;
- app imports should use a stable alias or package surface such as `@/ui` or `@scope/ui`;
- `entry` should point to the committed barrel using the path expected by the tool and current working directory;
- `componentSrcMap` may point to deep primitive or composed files, but only as metadata.

### Shadcn Common Layout

Typical source paths:

- `src/components/ui/*` for primitives;
- `src/components/patterns/*` or `src/components/shared/*` for composed reusable UI;
- `src/components/ui/index.ts` for primitive barrel;
- `src/design-system/index.ts` to re-export primitives and patterns for Claude Design.

Rules:

- keeping shadcn files in `src/components/ui/*` is fine;
- app, feature, shared, and pattern code should import primitives from `@/components/ui` when the repo chooses a public UI API;
- UI primitive files may keep file-level imports to avoid cycles;
- `src/design-system/index.ts` is appropriate when Claude Design needs both primitives and patterns;
- if the repo chooses file-level shadcn imports, do not pretend `src/components/ui/index.ts` is an enforced app boundary.

## Primitive And Pattern Boundaries

Keep the directory split explicit:

- `src/components/ui/*`, `src/ui/primitives/*`, or the repo's existing primitive path: base controls such as `Button`, `Badge`, `Input`, `Dialog`, `Tabs`, `Select`, `Tooltip`, `Popover`, `Card`.
- `src/components/shared/*`, `src/components/patterns/*`, `src/ui/components/*`, or the repo's established composed path: reusable composed UI such as `ProjectCard`, `WorkspaceHub`, dashboards, list rows, empty states, app shells, upload panels.
- `src/features/**`: feature-specific components that should not become Claude Design primitives by default.

If a composed component is useful to Claude Design previews, export it through the public barrel. If it is feature-specific, keep it in the feature and do not add it to the sync surface.

Default to documentation/config repairs. Only change component exports or directory structure when both are true:

- `rg` shows existing app or package usage that supports the proposed public surface.
- The repo's current architecture already has a stable boundary for that component type.

Do not create a new design-system barrel, move components, or split primitive/pattern directories just to satisfy a preview. If a preview asks for a component that has no public contract, either remove that preview dependency or document the missing contract.

## Config Rules

For `.design-sync/config.json`:

- `entry` should point to a committed source barrel, not `.design-sync/.ds-entry.tsx`, a generated cache file, or an ignored temporary shim.
- `componentSrcMap` may map public component names back to real source files for docs, previews, or review navigation. It is metadata, not the authority for imports.
- preview package names should reflect the public design-system surface, such as a workspace package name or a local alias declared by `.design-sync`.
- CSS and token entries should come from real production style files. Do not copy prototype CSS, HTML, fonts, assets, token names, or BEM classes from design artifacts into production.

Generated files can exist as tool output, especially under `.ds-sync/` or ignored paths. Repo conventions should identify which files are generated and which files are hand-maintained.

## Preview Import Rules

`.design-sync/previews/**` should import only the public surface exposed by the sync configuration.

Allowed shape:

```tsx
import { Button, ProjectCard } from "design-system-package";
```

Avoid these in previews:

```tsx
import { Button } from "@/components/ui/Button";
import { ProjectCard } from "../../src/components/patterns/ProjectCard";
import { Button } from "../.ds-entry";
```

If previews need a component that is not exported, first decide whether it belongs in the public design-system barrel. Do not patch previews with deep paths to bypass that decision.

## Documentation Rules

Keep `.design-sync` documentation current and generic to the repo:

- `conventions.md` should name the source-of-truth barrel, primitive directory, composed component directory, preview import package, token/style source, and forbidden prototype/deep-source imports.
- `NOTES.md` should record `/design-sync` operating rules: primitives and components must be exported through the public barrel, previews must not import deep source paths, generated files are not authoritative, and `componentSrcMap` is metadata only.
- Remove references to design artifact directories that are being retired or are no longer the current visual baseline.
- Do not document a prototype folder as an active contract after the repo has moved to a production design-system source.

## Fix Workflow

1. Find the real source of truth.
   Check existing app imports, package exports, barrels, `.design-sync/config.json`, and current git tracking status.

2. Classify the repo shape.
   Decide whether this is a monorepo/app-local design system, a shadcn common layout, a published UI package, or a pure file-level component folder.

3. Audit app imports before changing barrels.
   Count public-surface imports and deep primitive imports in app, feature, shared, and pattern layers. Use patterns like:

   ```bash
   rg -n "from ['\"]@/components/ui/" src
   rg -n "from ['\"]@/components/ui['\"]|from ['\"]@/ui['\"]|from ['\"]@/design-system['\"]" src
   rg -n "from ['\"]@/ui/" apps packages
   ```

   Report large deep-import counts and recommend a scoped migration plan. Do not rewrite primitive internals through the barrel.

4. Classify contract vs generated output.
   Treat committed `.design-sync` config, notes, conventions, and previews as source contracts. Treat `.ds-sync/` as generated local output unless the repo explicitly says otherwise.

5. Repair the `.design-sync` contract first.
   Point `.design-sync/config.json` at committed source paths. Remove prototype, generated, ignored, or deep-source paths from config and docs.

6. Rewrite previews to import only the public surface.
   If a preview needs a missing export, first decide whether the component already has a public contract. Add an export only when existing app usage and architecture support it; otherwise remove or narrow the preview.

7. Avoid structural changes by default.
   Split primitives from composed patterns, create barrels, or move components only when actual repo usage proves the current structure is leaking feature code or hiding an existing public UI contract.

8. Refresh `.design-sync/NOTES.md` and `.design-sync/conventions.md`.
   State the active contract, name generated directories such as `.ds-sync/`, and remove stale prototype references.

## Review Checklist

Before finishing, verify:

- `.design-sync/config.json` `entry` points at a committed real source file.
- all synced primitives and patterns are exported through the public barrel.
- app, feature, shared, and pattern code import the public UI barrel when the repo has chosen a public UI API.
- primitive-directory internals do not import through their own barrel.
- deep source imports are limited to barrels, component internals, or documented implementation exceptions.
- previews and app code converge on the same public component names.
- if existing app code has many deep imports, the final answer reports the count and recommends a scoped migration plan.
- previews do not import `src/**`, `@/**`, relative deep source paths, or `.ds-entry`.
- generated `.ds-sync/` or ignored tool-output files are not presented as source of truth.
- prototype CSS, HTML, fonts, assets, token names, and BEM classes were not copied into production source.
- docs name the current source of truth and do not mention retired design artifact directories as active inputs.
- `git diff --check` passes.
- if TypeScript barrels, preview imports, or package exports changed, the repo's available `typecheck` and `lint` commands pass.
- if the environment supports it, rerun `/design-sync` or the repo's preview build/smoke command after changing sync config or previews.
- if `/design-sync` cannot be run locally, state the blocker and the substitute verification used.
