## 测试、审查与交付

### 默认原则

- 先遵守仓库现有测试与交付命令，再补充建议
- 不要假定仓库一定有 Vitest、Playwright、Husky、Changesets
- 当仓库缺少测试基础设施时，先给出最小可落地方案

### 最小验证基线

- 类型检查通过
- lint 通过
- 改动涉及 UI 时，完成对应页面的浏览器验证或 smoke test
- 改动涉及核心流程时，补最接近风险的测试

### 测试优先级

1. feature hooks 的单元测试
2. screen 或共享组件的交互测试
3. 关键用户路径的 E2E

### 代码审查重点

- 页面是否仍然过重
- feature-local 组件是否错误放入 shared
- API 解包是否散落在页面中
- 新代码是否继续引入遗留样式体系
- 是否为了形式而过度抽 Hook 或过度 memo
- 表单是否符合“简单 state / 复杂 RHF+Zod”边界

### 交付建议

- 小步迁移，按 feature 逐步落地
- 每完成一个迁移阶段，都保持可验证和可回滚
- 文档优先记录：
  - 目录边界
  - 设计系统迁移边界
  - 状态与数据流边界
